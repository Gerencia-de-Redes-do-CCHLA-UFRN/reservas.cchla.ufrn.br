Estratégia de Ramificação (Branching): Main, Release e Dev

1. Arquitetura de Branches (O Modelo GitFlow Simplificado)

Para garantir a estabilidade do sistema que roda na infraestrutura do CCHLA, dividiremos o fluxo de trabalho em três esteiras principais. O objetivo é criar um "funil de qualidade" onde o código só chega à produção após ser validado.

  • main (Produção): É o código sagrado. Representa exatamente o que está rodando no servidor oficial (a sua VM Ubuntu). Ninguém nunca commita diretamente aqui. O código só chega na main após aprovação.
  • release (Homologação/Testes): É o ambiente de "pré-produção". Quando a branch dev atinge um estado maduro, ela é mesclada na release. Este código é enviado para testes. Se bugs forem encontrados aqui, são corrigidos antes de ir para a main.
  • dev (Desenvolvimento): É o coração do trabalho diário. Todas as novas funcionalidades, correções e experimentos são integrados nesta branch.

2. Passo a Passo: Reestruturando o Repositório Atual

Atualmente, todo o seu código está na main. Siga os passos abaixo no terminal da sua máquina (onde o projeto está clonado) para criar e publicar as novas ramificações no GitHub.

Passo 2.1: Sincronizar a máquina local com o GitHub

Certifique-se de que sua ramificação main local está 100% atualizada com o repositório remoto.

git checkout main
git pull origin main

Passo 2.2: Criar e publicar a branch DEV

Nós criamos a branch dev a partir do estado atual da main e a enviamos para o GitHub.

# Cria a branch dev e já muda para ela
git checkout -b dev

# Envia a nova branch para o GitHub (-u cria o vínculo upstream)
git push -u origin dev

Passo 2.3: Criar e publicar a branch RELEASE

Da mesma forma, criaremos a branch release a partir da dev.

# Cria a branch release a partir da dev
git checkout -b release

# Envia a branch release para o GitHub
git push -u origin release
Ajuste Obrigatório no GitHub Web:
Após enviar as branches, vá ao site do GitHub (github.com/redes-cchla-ufrn/formulario-reservas-cchla). Clique em Settings > Default branch. Mude a branch padrão de main para dev. Assim, todos os futuros Pull Requests apontarão automaticamente para a esteira de desenvolvimento, protegendo a produção.

3. O Fluxo de Trabalho Diário (Como trabalhar daqui para frente)

Agora que a arquitetura está montada, este será o seu fluxo operacional como administrador do sistema. Você não programará mais diretamente na main.

Cenário 1: Você vai criar uma nova funcionalidade (ex: Tela de Relatórios)

  1. Volte para a branch de desenvolvimento e atualize-a:
    git checkout dev
    git pull origin dev
  2. Crie uma branch temporária para a sua tarefa (Feature Branch):
    git checkout -b feature/tela-relatorios
  3. Trabalhe no código. Adicione os arquivos e faça o commit:
    git add .
    git commit -m "feat: adicionada nova tela de relatorios em pdf"
  4. Envie para o GitHub:
    git push origin feature/tela-relatorios
  5. Vá ao GitHub e abra um Pull Request (PR) da branch feature/tela-relatorios para a branch dev. Aprovação e Merge.

Cenário 2: Fechar uma versão e preparar para Produção (Lançamento)

A branch dev está cheia de recursos novos e você quer colocar no ar no servidor da UFRN.

  1. Vá ao GitHub e abra um Pull Request da branch dev para a branch release.
  2. Aprove o PR. A branch release agora contém as novidades.
  3. Faça o deploy da branch release em um ambiente de testes local ou servidor de homologação. Faça a auditoria. Tudo certo?
  4. No GitHub, abra o Pull Request Final da branch release para a branch main.
  5. Aprove o PR. Vá ao servidor Ubuntu da UFRN e puxe o código oficial:
    git checkout main
    git pull origin main
    docker compose restart (ou o seu comando de deploy)

Cenário 3: Emergência no Servidor! (Hotfix)

Há um bug crítico no formulário público que não pode esperar a esteira inteira de dev e release.

  1. Puxe uma branch diretamente da produção:
    git checkout main
    git checkout -b hotfix/erro-formulario
  2. Conserte o erro, faça o commit e o push.
    git add . && git commit -m "fix: correção de erro critico no form"
    git push origin hotfix/erro-formulario
  3. No GitHub, abra um PR da hotfix/erro-formulario direto para a main e lance no servidor.
  4. Importante: Não esqueça de fazer o merge deste hotfix também para a dev, senão o erro voltará no próximo lançamento comum!

4. Resumo de Proteção de Ramificação (Branch Protection Rules)

Para garantir que esse manual seja seguido mesmo se outros técnicos do CCHLA entrarem no projeto, configure as proteções no GitHub:

  • Acesse Settings > Branches > Add branch ruleset.
  • Alvo: main.
  • Ative: Require a pull request before merging (Não permite commit direto na main).
  • Ative: Require approvals (Exige que pelo menos uma pessoa revise o código).
  • Faça a mesma regra de proteção para a branch release.
