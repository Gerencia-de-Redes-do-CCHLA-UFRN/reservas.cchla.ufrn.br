DOCUMENTAÇÃO TÉCNICA UNIFICADA: SISTEMA DE GESTÃO DE RESERVAS CCHLA

1. Engenharia de Software e Análise de Requisitos

1.1. Visão Geral e Processos de Negócio

O Sistema de Gestão de Reservas CCHLA foi projetado para modernizar, auditar e proteger o agendamento de espaços físicos. O processo de negócio central consiste em receber solicitações da comunidade acadêmica, cruzar essas solicitações com bloqueios institucionais (manutenções e feriados) e fornecer à secretaria uma interface administrativa de aprovação, blindada por uma arquitetura Zero-Trust (Serverless).

1.2. Atores do Sistema

  • Solicitante (Público): Membro da comunidade (aluno, professor, técnico). Não possui credenciais de login. Interage com o sistema exclusivamente para enviar pedidos, consultar o calendário visual e rastrear seu Ticket.
  • Secretaria (Administrador): Operador do sistema. Possui login protegido por JWT. Tem privilégios para aprovar, reprovar, criar e excluir reservas, além de gerenciar bloqueios e manutenções.
  • Sistema (Backend): Ator invisível responsável por validar cruzamentos de horários, assinar ações de auditoria e barrar requisições não autorizadas.

1.3. Requisitos Funcionais (RF)

  • RF01: O sistema deve permitir que solicitantes enviem pedidos de reserva únicos, semanais ou personalizados.
  • RF02: O sistema deve gerar um Ticket único (padrão Mercosul) para agrupar múltiplas reservas da mesma solicitação.
  • RF03: A secretaria deve ter um painel para visualizar, aprovar ou reprovar reservas em lote ou individualmente.
  • RF04: O sistema deve alertar a secretaria caso uma nova aprovação entre em conflito de horário com uma reserva previamente aprovada ou feriado.
  • RF05: O sistema deve fornecer uma rota pública para acompanhamento de status via Ticket.
  • RF06: O sistema deve desconectar automaticamente usuários ociosos após 30 minutos.
  • RF07: O sistema deve restringir as reservas de auditórios ao intervalo estrito das 06:00 às 23:00.
  • RF08: O backend deve agrupar novas solicitações sob um mesmo Ticket caso pertençam ao mesmo solicitante (Matrícula + E-mail ou Telefone).
  • RF09: O frontend deve exibir dinamicamente os blocos de horários livres residuais do dia selecionado em caso de colisão temporal.
  • RF10: O sistema deve barrar de forma intransponível qualquer tentativa de reserva que colida com bloqueios de manutenção ativa.

1.4. Requisitos Não Funcionais (RNF)

  • RNF01: A arquitetura deve ser Serverless (Google Cloud Functions / Firebase).
  • RNF02: A segurança de rotas administrativas deve utilizar autenticação via JWT (JSON Web Token), sem exposição de SDK no frontend.
  • RNF03: O frontend deve ser distribuído em múltiplos sites (Multisite Hosting) para isolamento de aplicações.
  • RNF04: O banco de dados deve ser NoSQL orientado a documentos (Firestore).
  • RNF05: Credenciais e chaves de API não devem ser trafegadas ou expostas no código cliente (Frontend).

1.5. Fluxo do Sistema (User Flow)

  • Fluxo de Solicitação: Usuário acessa formulário → Preenche dados → Adiciona locais e horários ao "carrinho" → Resolve conflitos apontados pelo sistema → Submete → Recebe Ticket e link de WhatsApp.
  • Fluxo de Avaliação: Secretaria faz login → Backend valida JWT → Painel carrega dados → Secretaria visualiza alertas de colisão → Secretaria marca como Aprovado/Reprovado (com motivo) → Backend assina a ação (IP/Email) e grava no banco.
  • Fluxo de Acompanhamento: Usuário acessa tela de acompanhamento → Digita o Ticket → Backend público filtra e devolve dados essenciais → Usuário visualiza o status.

2. Banco de Dados (Firestore NoSQL)

2.1. Características Gerais

O banco de dados opera em ambiente Serverless no Google Cloud (Firestore). A estrutura é não-relacional, otimizada para alta velocidade de leitura (Read-Heavy). O design utiliza uma única coleção "flat" (plana), eliminando a necessidade de "Joins" complexos e reduzindo custos operacionais.

3. Backend (Cloud Functions / Node.js)

3.1. Visão Geral

O backend atua como uma API RESTful Serverless. Ele intercepta todas as requisições HTTP do frontend, resolve as políticas de CORS (Cross-Origin Resource Sharing) e aplica regras rigorosas de segurança através do firebase-admin antes de permitir operações no Firestore.

3.2. Detalhamento de Funções e Métodos

  • validarTokenAdmin(req) [Middleware Interno]: Extrai o cabeçalho HTTP Authorization: Bearer. Utiliza o Firebase Admin SDK para descriptografar e validar o JWT. Rejeita sem token com 401.
  • fazerLogin [POST]: Rota pública de autenticação. Comunica-se Server-to-Server com a API do Identity Toolkit do Google usando a chave IDENTITY_API_KEY.
  • checarDisponibilidade [POST]: Devolve os arrays conflitos e agendaCompleta, permitindo ao cliente mapear lacunas livres residuais.
  • enviarSolicitacao [POST]: Valida o reCAPTCHA. Varre o banco buscando correspondência de Matrícula + Email para reaproveitar Tickets. Executa gravações atômicas em lote (db.batch()).
  • obterEventosCalendario [GET]: Rota pública otimizada que recupera exclusivamente documentos com Status Aprovado, omitindo dados confidenciais.
  • adminObterReservas [GET]: Rota protegida. Lê todo o banco de dados. A ordenação cronológica foi programada na memória RAM do Node.js para garantir que dados legados sem timestamp não sejam omitidos.
  • adminAtualizarDocumento [POST]: Rota protegida. Injeta automaticamente, pelo lado do servidor, o email do usuário logado no campo Status_Alterado_Por.
  • adminCadastrarDocumento [POST]: Rota protegida. Insere carimbos de tempo (serverTimestamp) e a assinatura da secretaria em bloqueios de feriados e manutenções diretas.
  • adminExcluirDocumento [POST]: Rota protegida. Contém uma trava rígida (*Hardcoded Privilege*): rejeita a operação com status 403 se o e-mail do token for diferente da credencial de root configurada.
  • consultarTicket [POST]: Rota pública. Executa Query exata filtrando pelo Ticket recebido em maiúsculas e devolve apenas os campos essenciais de acompanhamento.

4. Frontend (Aplicações Multisite)

4.1. Visão Geral

Composto por microsserviços estruturados em diretórios modulares independentes rodando Vanilla JavaScript (ES6+). O ecossistema é agnóstico ao banco de dados e não possui SDK do Firebase configurado no lado do cliente, operando via fetch.

4.2. Especificidades das Aplicações

  • Formulário Público: Controla intenções em memória RAM via listaDeReservasGeral. Consome checarDisponibilidade em tempo real. Implementa travas lógicas e união de intervalos para calcular sugestões de horários livres.
  • Painel Administrativo: Exige token no sessionStorage. Implementa renderização adaptativa em lote (Batch Rendering), agrupando registros pendentes do mesmo solicitante. Inclui Error Logger que renderiza diagnósticos na interface em caso de falhas críticas.
  • Calendário: Inicializa o FullCalendar. A função formatarDataCustom mitiga erros de fuso horário (Timezone Shift), processando as strings ISO textualmente para exibir Feriados na data exata, além de separar blocos allDay da grade horária.
  • Acompanhamento: Consome a rota pública consultarTicket. Lê Query Parameters (?ticket=ABC1D23) para abertura direta. Suporta tooltips flutuantes para exibir motivos de reprovação ao hover.
  • Funções Globais: iniciarRastreamentoInatividade() reinicia o timer de 30 minutos a cada input e desloga usuários ociosos. formatarDataHoraCompleta() padroniza as respostas de data vindas do NoSQL.

5. Infraestrutura e Sustentação

5.1. Roteiro Histórico de Desafios e Sucessos

  1. Fase 1: Auditoria e Quebra de Client-Side SDK
    Problema: Chaves expostas no HTML permitiam manipulação do banco via console.
    Sucesso: Transferência de todo o pipeline de negócios para as Cloud Functions. O frontend passou a atuar como um cliente REST cego.
  2. Fase 2: Implantação de Login API
    Problema: Rejeição de inicialização por uso de prefixos reservados pelo Google (FIREBASE_API_KEY).
    Sucesso: Isolamento das chaves sob IDENTITY_API_KEY no arquivo .env.
  3. Fase 3: Estabilização do Painel (CORS e Limitações NoSQL)
    Problema: Bloqueio 401/403 (Preflight OPTIONS). O filtro orderBy do banco omitia dados legados. A injeção de HTML travava a inicialização.
    Sucesso: Diretiva allUsers e resposta 204 ao OPTIONS. Ordenação cronológica realocada para a memória do Node.js. Listeners injetados em DOMContentLoaded.
  4. Fase 4: Conserto Visual do Calendário (Timezone Shift)
    Problema: Feriados retrocediam 3h (GMT-3) do new Date().
    Sucesso: Reescrita do pipeline de leitura, isolando o parseamento das strings via split('T') ignorando o client timezone.

6. Manual de Deploy Geral do Sistema

6.1. Estrutura de Diretórios Exigida

A topologia modular exige que o repositório local mantenha a seguinte árvore para não quebrar referências relativas:

📦 Raiz do Projeto
 ┣ 📂 app-acompanhamento
 ┣ 📂 app-calendario
 ┣ 📂 app-formulario
 ┣ 📂 app-manutencao
 ┣ 📂 app-painel
 ┣ 📂 functions
 ┃ ┣ 📜 index.js
 ┃ ┗ 📜 .env
 ┣ 📜 .firebaserc
 ┗ 📜 firebase.json

6.2. Comandos de Implantação

Para implantar, utilize a Interface de Linha de Comando do Firebase (firebase-tools):

  • Deploy Global (Backend e todos os Frontends):
    firebase deploy
  • Deploy Exclusivo de Frontend (Multisite):
    firebase deploy --only hosting
  • Deploy Granular por Aplicação:
    firebase deploy --only hosting:painel
    firebase deploy --only hosting:formulario-reservas
    firebase deploy --only hosting:calendario
    firebase deploy --only hosting:acompanhamento
  • Deploy Exclusivo de Regras de Negócio (Backend):
    firebase deploy --only functions
