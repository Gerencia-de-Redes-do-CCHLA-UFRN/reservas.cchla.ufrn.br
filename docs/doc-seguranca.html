Documentação de Segurança

Análise das camadas de proteção do Sistema de Reservas de Auditórios do CCHLA/UFRN

1. Matriz de Ameaças e Mitigações

Alta

Acesso não autorizado ao painel

Mitigado por: JWT com expiração de 24h, senhas com bcrypt, rate limit de login, log de tentativas.

Alta

Injeção no MongoDB

Mitigado por: Mongoose com schemas tipados, validação de campos no controller. Strict: false apenas para compatibilidade legado.

Alta

Spam / Flood de solicitações

Mitigado por: reCAPTCHA no formulário público, limite de 50 itens por requisição.

Média

Exposição de credenciais no repositório

Mitigado por: .env no .gitignore, .env.example sem valores reais, serviceAccountKey.json ignorado.

Média

Cross-Origin (CORS) mal configurado

Mitigado por: CORS restrito a origens específicas (reservas.cchla.ufrn.br e http://localhost).

Média

Acesso ao banco via mongo-express

Mitigado por: Restrição de IP no Nginx (127.0.0.1 e 10.0.0.0/8), autenticação basic auth.

Baixa

Vazamento de dados via calendário público

Mitigado por: Calendário expõe apenas reservas com status "Aprovado". Dados sensíveis como telefone e e-mail não são retornados.

Baixa

Falsificação de solicitação (CSRF)

Mitigado por: Formulário público usa reCAPTCHA. Rotas admin usam JWT via header (não cookies), imune a CSRF clássico.

Baixa

Man-in-the-Middle (MitM)

Mitigado por: Produção em HTTPS via proxy reverso (nginx). SMTP com STARTTLS (porta 587).

Info

Ataque de força bruta no login

Mitigado por: Bloqueio de 5 tentativas falhas em 15 minutos. Log de todas as tentativas no SecurityLog.

3. Fluxo de Segurança no Login

Usuário → POST /api/fazerLogin { email, senha }
  │
  ├─ 1. Valida campos obrigatórios
  ├─ 2. Verifica rate limit (≤ 5 falhas em 15 min)
  │     └─ Se excedido → 403 "bloqueado temporariamente"
  ├─ 3. Busca admin no MongoDB por e-mail
  │     └─ Se não encontrado → 401 + SecurityLog { success: false, userExists: false }
  ├─ 4. Compara senha com bcrypt
  │     └─ Se inválida → 401 + SecurityLog { success: false, userExists: true }
  ├─ 5. Gera JWT { id, email, nivel } com expiração 24h
  ├─ 6. SecurityLog { success: true }
  └─ 7. Retorna { sucesso: true, token }

4. Fluxo de Segurança nas Solicitações

Usuário → POST /api/enviarSolicitacao { reservas[], recaptchaToken }
  │
  ├─ 1. Verifica se reCAPTCHA token está presente
  ├─ 2. Valida reCAPTCHA com Google (se RECAPTCHA_SECRET configurada)
  ├─ 3. Valida array de reservas (≤ 50 itens)
  ├─ 4. Para cada item:
  │     ├─ Campos obrigatórios presentes
  │     ├─ hIni < hFim (fraude temporal)
  │     └─ Horários dentro de 07:00–21:30
  ├─ 5. Busca histórico do solicitante por matrícula
  │     └─ Reusa ticket se já existir
  ├─ 6. Insere no MongoDB via insertMany
  └─ 7. Dispara e-mail de confirmação (falha não bloqueia)

6. Recomendações Futuras

  1. Helmet.js — Adicionar middleware helmet para headers de segurança (X-Content-Type-Options, X-Frame-Options, CSP).
  2. HTTPS forçado — Redirecionar HTTP → HTTPS no nginx (atualmente depende do proxy externo).
  3. Validação com express-validator — Substituir validações manuais por schema validation nos controllers.
  4. Refresh token — Implementar refresh token para evitar exposição prolongada do JWT.
  5. GeoIP no SecurityLog — Integrar geolocalização nos logs de segurança (campo country já previsto no schema).
  6. Revogar serviceAccountKey.json — A chave Firebase ainda existe no filesystem; deve ser revogada no GCP IAM.
  7. Testes de penetração — Realizar scan com OWASP ZAP ou similar.
