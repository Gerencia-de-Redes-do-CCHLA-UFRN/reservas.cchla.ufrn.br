Documentação de Segurança da Informação

Políticas Zero-Trust, Sanitização de Payload e Mitigação de Vulnerabilidades

1. Visão Geral da Arquitetura de Defesa

O ecossistema do CCHLA adota o princípio de segurança Zero-Trust (Confiança Zero). Este paradigma determina que o servidor (Backend em Cloud Functions) jamais deve confiar nos dados enviados pelo navegador do usuário (Frontend). Qualquer requisição que atinge o banco de dados Firestore é considerada hostil até que seja criptograficamente validada e sanitizada.

2. Firewalls e Políticas de Acesso (CORS)

Vulnerabilidade Mitigada: Requisições Forjadas de Origens Desconhecidas
A ausência de políticas estritas de CORS permitiria que domínios externos (ou sites maliciosos) enviassem tráfego diretamente para a API de reservas, consumindo cotas de faturamento da nuvem e poluindo o banco de dados.

Para garantir que apenas os formulários legítimos institucionais comuniquem-se com o servidor, implementou-se uma Whitelist (Lista de Permissões) de origens. O módulo cors foi configurado com um interceptador dinâmico que rejeita conexões de IPs e URLs não mapeadas.

3. Sanitização de Payload (Validação Server-Side)

O Frontend do sistema contém validações visuais em HTML/JS, mas elas são facilmente contornáveis por usuários com conhecimentos técnicos (via ferramentas como *cURL* ou *Postman*). Para proteger a integridade do banco de dados, o Endpoint enviarSolicitacao implementa filtros rigorosos.

Camadas de Sanitização Implementadas:
  • Validação de Tipo e Estrutura: Rejeita requisições onde a propriedade reservas não seja uma Array válida ou esteja vazia, prevenindo falhas de Crash no iterador do Node.js (Erro 500).
  • Proteção Anti-Flood (Limite de Lote): Bloqueia requisições contendo mais de 50 agendamentos simultâneos na mesma transação, impedindo exaustão de armazenamento.
  • Prevenção de Fraude Temporal: Valida matematicamente as variáveis de tempo. A requisição é derrubada instantaneamente se a hora de término (hFim) for inferior ou igual à hora de início (hIni).
  • Conformidade Operacional: Aplica a trava institucional no servidor garantindo que horários solicitados estejam estritamente dentro da matriz das 06:00 às 23:00.

Implementação Lógica no Backend

// Trecho de Sanitização Crítica na Cloud Function
if (!payload.reservas || !Array.isArray(payload.reservas) || payload.reservas.length === 0) {
    throw new Error("O pacote de reservas está vazio ou em formato inválido.");
}
if (payload.reservas.length > 50) {
    throw new Error("Limite excedido (Proteção contra Flood).");
}
for (let item of payload.reservas) {
    if (item.hIni >= item.hFim) {
        throw new Error("Fraude temporal: Hora de término inferior à inicial.");
    }
    if (item.hIni < "06:00" || item.hFim > "23:00") {
        throw new Error("Horário fora da matriz operacional (06:00 às 23:00).");
    }
}

4. Autenticação e Proteção contra Automação

4.1. Verificação S2S do reCAPTCHA

O sistema não confia na presença superficial de um token. O Node.js se comunica diretamente com a API do Google (Server-to-Server) utilizando a RECAPTCHA_SECRET oculta nas variáveis de ambiente. Se a pontuação de risco da Google determinar que a requisição partiu de um Bot (Robô), a operação no Firestore é cancelada preventivamente.

4.2. Trilha de Auditoria Forense Inviolável

Em operações de Secretaria (Aprovação, Exclusão, Alteração de Status), o sistema ignora qualquer identificador enviado pelo navegador. O Backend extrai o e-mail do autor da ação diretamente da descriptografia do token JWT e injeta compulsóriamente no campo Status_Alterado_Por. Isso impossibilita que um operador forje a identidade de outro na trilha de auditoria.

4.3. Trava "Root" Administrativa (Hardcode Privilege)

O Endpoint adminExcluirDocumento promove a deleção definitiva de registros. Para evitar que contas operacionais comuns (ou contas comprometidas) apaguem o histórico do CCHLA, existe uma trava em nível de código (Hardcode) que exige que a conta originadora seja a credencial mestre do administrador tobias.rocha@ufrn.br. Qualquer outra conta administrativa autenticada receberá o bloqueio HTTP 403 Forbidden.

5. Higienização de Saída de Dados (Prevenção de Vazamentos PII)

As rotas públicas obterEventosCalendario e consultarTicket atuam sob os princípios da LGPD (Lei Geral de Proteção de Dados). Antes de despachar os documentos JSON do Firestore para a internet, a API atua como um funil filtrante, criando novos objetos em RAM contendo estritamente os campos necessários para renderização visual (Local, Data, Status, Atividade), podando informações confidenciais como E-mails e Telefones de Contato.
