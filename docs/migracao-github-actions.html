Automação de Deploy (CI/CD)

Integração do GitHub Actions com a VM Ubuntu (On-Premise UFRN)

Estratégia: Self-Hosted Runner (Bypass Seguro de Firewall Institucional)

Porquê um Self-Hosted Runner?
Servidores universitários (como os da UFRN) possuem regras estritas de firewall de entrada (Inbound). Se os servidores externos do GitHub tentarem conectar-se via SSH à sua VM, a ligação será bloqueada. O Self-Hosted Runner é um pequeno serviço instalado na sua VM que comunica com o GitHub de "dentro para fora" (Outbound), contornando o bloqueio de forma totalmente segura e aprovada por equipas de infraestrutura.

1. Instalar o Agente (Runner) na VM da UFRN

O primeiro passo é registar a sua Máquina Virtual no GitHub. Aceda ao repositório no seu navegador (github.com/redes-cchla-ufrn/formulario-reservas-cchla) e vá a:

  • Settings (Configurações) > Actions > Runners
  • Clique no botão verde New self-hosted runner.
  • Selecione Linux e a arquitetura x64.

O GitHub irá fornecer-lhe uma série de comandos exatos. Abra o terminal SSH da sua VM Ubuntu e execute-os. Serão algo muito parecido com isto:

# Criar uma pasta para o agente (Não executar como root!)
mkdir actions-runner && cd actions-runner

# Descarregar o pacote de instalação
curl -o actions-runner-linux-x64-2.316.1.tar.gz -L https://github.com/actions/runner/releases/download/v2.316.1/actions-runner-linux-x64-2.316.1.tar.gz

# Extrair
tar xzf ./actions-runner-linux-x64-2.316.1.tar.gz

# Configurar o agente (cole a chave exata fornecida no site do GitHub)
./config.sh --url https://github.com/redes-cchla-ufrn/formulario-reservas-cchla --token SEU_TOKEN_GERADO_PELO_GITHUB

1.1. Instalar o Agente como Serviço de Background

Para garantir que o agente inicia automaticamente se a VM for reiniciada, instale-o como um serviço do sistema (systemd):

sudo ./svc.sh install
sudo ./svc.sh start
sudo ./svc.sh status

2. Configurar Permissões do Docker na VM

O agente do GitHub corre sob um utilizador próprio no Ubuntu e precisará de permissões para executar o comando docker compose sem pedir senha de administrador (sudo).

No terminal da sua VM, execute:

# Assumindo que o utilizador do seu runner se chama "ubuntu" ou "tobias"
sudo usermod -aG docker $USER

# Se o runner criar um utilizador específico chamado "runner", execute:
# sudo usermod -aG docker runner

# Reinicie o serviço do Docker para aplicar
sudo systemctl restart docker

3. Criar os Ficheiros de Workflow no Código

Agora vamos ensinar ao GitHub o que fazer quando aprovar um Pull Request para a main (Produção) ou release (Homologação). Na pasta do seu projeto local (no seu computador), crie um diretório chamado exato .github/workflows/.

3.1. Workflow de Produção (Deploy da branch MAIN)

Crie um ficheiro chamado deploy-producao.yml dentro da pasta .github/workflows/.

name: Deploy Produção (UFRN)

on:
  push:
    branches:
      - main # O gatilho dispara sempre que houver um merge para a main

jobs:
  deploy:
    # A diretiva self-hosted diz ao GitHub para procurar a VM da UFRN
    runs-on: self-hosted

    steps:
      - name: 📥 Fazer o Checkout do Código
        uses: actions/checkout@v4

      - name: 🚀 Sincronizar e Reiniciar Docker Compose
        run: |
          # Navega para a pasta do sistema em produção
          cd /opt/sistema-reservas
          
          # Puxa o código mais recente do GitHub
          git pull origin main
          
          # Reconstrói as imagens e reinicia os contentores sem tempo de inatividade
          docker compose up -d --build
          
          # Limpa imagens antigas órfãs para poupar espaço em disco na VM
          docker image prune -f

3.2. Workflow de Homologação (Deploy da branch RELEASE)

É uma excelente prática ter uma cópia exata do sistema a correr noutra porta (ex: porta 8080) para testar novidades antes de afetar os utilizadores reais. Se tiver uma pasta /opt/sistema-reservas-homolog, crie o ficheiro deploy-homologacao.yml:

name: Deploy Homologação

on:
  push:
    branches:
      - release

jobs:
  deploy:
    runs-on: self-hosted

    steps:
      - name: 📥 Fazer o Checkout do Código
        uses: actions/checkout@v4

      - name: 🧪 Atualizar Ambiente de Homologação
        run: |
          cd /opt/sistema-reservas-homolog
          git pull origin release
          docker compose up -d --build

4. O Ciclo de Vida Automatizado (Como tudo funciona agora)

A partir deste momento, o seu trabalho manual no servidor Ubuntu acabou. O fluxo será o seguinte:

  1. Trabalha tranquilamente na branch dev no seu computador.
  2. Quando estiver pronto para testar, abre um Pull Request de dev para release no GitHub e clica em Merge.
  3. O GitHub Actions acorda instantaneamente, avisa o agente instalado na VM da UFRN, o agente entra na pasta de homologação, puxa o código e constrói o Docker sozinho.
  4. Depois de testar na homologação e confirmar que não há bugs, abre o último Pull Request de release para main.
  5. No segundo em que aprovar o Pull Request, o servidor da UFRN puxará o código e reiniciará o Nginx e a API do Node.js automaticamente. A nova versão estará em produção (Live) em menos de 10 segundos.
Monitorização e Logs:
Pode acompanhar a execução do deploy em tempo real no site do GitHub. Basta clicar no separador "Actions" no topo do seu repositório. Ocorreu um erro na compilação do Docker? O GitHub mostrará o log exato do terminal do Ubuntu diretamente no seu ecrã, dispensando a necessidade de aceder via SSH para descobrir o problema.
