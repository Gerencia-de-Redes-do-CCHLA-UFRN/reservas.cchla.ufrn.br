1. Engenharia de Software e Análise de Requisitos

1.1. Visão Geral e Processos de Negócio

O Sistema de Gestão de Reservas CCHLA foi arquitetado para automatizar, rastrear e proteger o agendamento de infraestrutura física da instituição. O fluxo de negócio baseia-se na captação de demandas acadêmicas, cruzamento em tempo real com matrizes de bloqueio (manutenções preventivas, feriados e colisão de turmas) e deliberação via interface administrativa blindada sob o paradigma Zero-Trust.

1.2. Atores do Sistema

  • Solicitante (Público): Representado pela comunidade acadêmica. Interage com a camada pública do sistema exclusivamente para submissão de intenções de reserva, visualização da grade ocupacional no calendário e acompanhamento de protocolo (Ticket).
  • Secretaria (Administrador): Operador interno. Possui acesso restrito protegido por JSON Web Tokens (JWT). Detém autonomia global para homologar, rejeitar, criar, retificar e excluir registros, além de gerir paralisações de infraestrutura.
  • Engine Serverless (Backend): Camada lógica invisível. Responsável por arbitrar regras de negócio, assinar trilhas de auditoria forense, proteger a base de dados contra scraping (raspagem) e invocar rotinas de disparo de correio eletrônico.

1.3. Requisitos Funcionais (RF)

ID Descrição do Requisito
RF01O sistema deve captar reservas na modalidade única, recorrente semanal e recorrência personalizada com salto de semanas.
RF02O motor lógico deve agrupar requisições isoladas oriundas do mesmo solicitante sob um identificador transacional único (Ticket Mercosul).
RF03A interface administrativa deve consolidar solicitações pendentes atreladas ao mesmo autor para viabilizar homologações e indeferimentos em lote (Batch Processing).
RF04O sistema deve emitir avisos de nível Warning caso uma homologação concorra com reservas aprovadas, e nível Critical (bloqueio de submissão) caso concorra com manutenções de infraestrutura ativas.
RF05Deve existir um módulo de consulta de acesso público capaz de ler parâmetros de URL (Query Strings) para injeção automática do número do Ticket.
RF06Sessões administrativas devem possuir tempo de vida ocioso máximo de 30 minutos (Inactivity Timeout), com aviso prévio em Modal antes da destruição da sessão.
RF07A matriz de tempo para qualquer auditório deve estar restrita ao escopo operacional das 06:00 às 23:00.

1.4. Requisitos Não Funcionais (RNF)

ID Descrição do Requisito
RNF01A arquitetura geral do sistema deve ser obrigatoriamente Serverless, suportada por provedores de nuvem em contêineres sob demanda (Google Cloud Run).
RNF02O frontend não deve armazenar credenciais estáticas de banco de dados, atuando exclusivamente como um cliente HTTP/REST cego.
RNF03O banco de dados deve utilizar o paradigma NoSQL orientado a documentos para sustentar picos de leitura rápida (Read-Heavy).
RNF04A aplicação cliente deve ser fracionada em múltiplos domínios (Firebase Hosting Multisite) visando isolamento de falhas, independência de cache e organização modular de rotas.

1.5. Modelagem de Fluxos (User Flow)

Fluxo Primário (Submissão): Acesso à SPA de Formulário → Preenchimento de perfil → Construção de matrizes de local/tempo → Avaliação passiva de colisões residuais pelo backend → Confirmação ReCAPTCHA → Submissão HTTP POST → Recepção do Ticket de Protocolo.

Fluxo Secundário (Triagem): Autenticação Administrativa → Captura de JWT → Carregamento de Cache Completo do Banco → Processamento em RAM de Agrupamentos por Ticket → Avaliação Humana de Alertas Sintéticos → Emissão de Decisão (Aprovação/Reprovação) → Gravação de Metadados de Auditoria.

2. Modelagem do Banco de Dados (Firestore)

2.1. Escolha Arquitetural

A persistência de dados foi implementada no Google Cloud Firestore. A natureza NoSQL foi selecionada em virtude da ausência de rigidez estrutural, permitindo a evolução de atributos sem a necessidade de migrações pesadas de esquemas (Schema Migrations) comuns em bancos relacionais como PostgreSQL ou MySQL. O design utiliza um modelo Flat (única coleção principal) para evitar as limitações de junções (Joins).

2.2. Dicionário de Dados da Coleção: reservas

3. Especificação da API e Backend (Cloud Functions)

3.1. Arquitetura da API REST

O backend, escrito em Node.js (Ambiente Node 18+), atua como barreira de contenção. A integração de rotas (Endpoints) é despachada através de contêineres independentes no Google Cloud Run, respondendo a gatilhos HTTP puros.

3.2. Funções e Endpoints Documentados

  • validarTokenAdmin(req) [Middleware de Segurança]
    Rotina interna transversal. Intercepta o req.headers.authorization. Quebra o prefixo "Bearer " e submete o hash ao módulo admin.auth().verifyIdToken(). Exceções criptográficas, assinaturas adulteradas ou tokens expirados disparam erro estrito HTTP 401 Unauthorized com drop de pacote imediato.
  • fazerLogin [POST]
    Abertura de túnel OAUTH. Captura JSON contendo credenciais nativas. Instancia conexão S2S (Server-to-Server) contra os servidores do Google Identity Toolkit via `https://identitytoolkit.googleapis.com/v1/accounts:signInWithPassword`. Retorna ao frontend um payload contendo o idToken seguro.
  • checarDisponibilidade [POST]
    Motor de interseção de calendário. Aceita arrays de pacotes de dados temporais recém-construídos pelo usuário no frontend. Executa Queries limitadas na janela temporal D-2 a D+2 do alvo para mitigar gargalos de I/O. Retorna as matrizes conflitos e agendaCompleta.
  • enviarSolicitacao [POST]
    Endpoint de finalização de pedido público. Valida a integridade do desafio Google reCAPTCHA. Utiliza a instrução db.batch() do Firestore para aplicar escritas em lote atômicas (garantindo que se um registro da lista falhar por timeout, o lote inteiro sofre Rollback). Invoca webhook HTTP auxiliar no App Script da reitoria para despacho de confirmação SMTP.
  • obterEventosCalendario [GET]
    Rota pública de leitura higienizada. Executa busca ampla apenas em documentos filtrados com Status == 'Aprovado'. Mapeia e omite propriedades de PII (Personally Identifiable Information) como e-mail, telefone e IP antes de serializar o JSON, garantindo aderência rigorosa a políticas de proteção de dados (LGPD).
  • adminObterReservas [GET]
    Extrator integral da base. Exige JWT. Decisão de Engenharia Crítica: A rotina omite a cláusula de busca .orderBy() na requisição direta ao Firestore, varrendo toda a coleção sem filtros nativos. A ordenação decrescente baseada no timestamp é executada nativamente pela máquina virtual V8 do Node.js através de `Array.sort()`. Isso assegura que documentos legados desprovidos de carimbo de tempo não sejam vítimas de omissão de índice.
  • adminAtualizarDocumento [POST]
    Responsável por transições de estado (Aprovar/Reprovar/Corrigir). Extrai silenciosamente o email do payload validado no JWT e injeta compulsóriamente no atributo Status_Alterado_Por junto de Status_Alterado_Data.
  • adminCadastrarDocumento [POST]
    Canal de injeção direta de sistema (Bypass de triagem). Autoconcebe a propriedade Status: "Aprovado".
  • adminExcluirDocumento [POST]
    Endpoint de aniquilação destrutiva. Exige JWT. Aplica Privilege Hardcoding, checando explicitamente se o e-mail validado no decodificador é correspondente à credencial de root configurada na instituição. Submissões forjadas por contas legítimas não-autorizadas resultam em HTTP 403 Forbidden.
  • consultarTicket [POST]
    Motor de consulta pública. Aplica coerção algorítmica (`toUpperCase()`) no parâmetro injetado pelo usuário e executa varredura do índice "Ticket". Retorna subset filtrado do documento original.

4. Especificação do Frontend (Multisite Application)

4.1. Fundamentos da Camada Cliente

As interfaces foram construídas sob a égide do "Vanilla JavaScript" puro (ES6+), desprovidas de bloatwares como React ou Angular, garantindo métricas de Time-To-Interactive (TTI) na casa dos milissegundos. Toda a lógica de estado do SDK do Firebase foi removida do código executado no navegador, forçando as aplicações a operarem como clientes isolados que gerenciam seus próprios pacotes HTTP.

4.2. Tratamento Global de Ciclos de Vida

  • Sessões (Zero-Trust): O manuseio de segurança é atrelado exclusivamente à Storage local do navegador (sessionStorage). Fechar a aba acarreta na aniquilação criptográfica da prova de identidade.
  • Temporal Dead Zone (TDZ): Todas as chamadas de inicialização de UI e Fetch API foram encapsuladas em listeners de DOMContentLoaded. Esta contenção previne cenários de "Silent Crash" onde o Javascript tentaria manipular seletores de DOM e tags DIV não resolvidas na árvore de renderização do Webkit.
  • Inactivity Engine: O módulo varre inputs de interface e engatilha a cadeia de Timeout usando setTimeout. Eventos passivos de rolar de tela ou mover o ponteiro anulam o relógio, garantindo proteção sem impactar UX.

4.3. Detalhamento por Módulo (Micro-Frontends)

  • Formulário Público (app-formulario):
    Opera um array virtual atuando como um "carrinho de compras" de tempo. A conversão de pedidos semanais/personalizados engatilha um iterador matemático (Date Cursor) que projeta instâncias diárias incrementais, convertendo uma intenção macro em micro-registros isolados, submetidos em bloco. Emprega heurísticas de intersecção em matrizes de tempo para sugerir ao usuário períodos vagos em caso de choque no calendário.
  • Painel Administrativo (app-painel):
    Aplica rotinas pesadas de ordenação e Batch Rendering (Renderização em Lote). Ao receber as matrizes do servidor, reduz os dados unindo entradas pendentes pelo atributo Solicitante e exibe um cartão condensado contendo múltiplos Checkboxes/Radios, promovendo processamento sistêmico acelerado. Inclui um Logger de Exceções que desenha o stack-trace de falhas na porta 443 visualmente no HTML.
  • Calendário de Ocupação (app-calendario):
    Incorpora a engine de terceiros FullCalendar. A importação sofre mutação de comportamento para mitigar Timezone Shifts: Em vez de permitir a serialização nativa da biblioteca (que abatia 3 horas da exibição de feriados via GMT-3), o parser extrai as propriedades absolutas da string ISO via método split('T'), garantindo sincronia perfeita com matrizes do banco independente do país do terminal que requisita a página.
  • Acompanhamento (app-acompanhamento):
    Rota flexível desenhada para integrar-se ao App Script de reitoria. Executa parseamento do objeto window.location.search interceptando variáveis de query string. Converte entradas passivas do tipo ?ticket=AAA00 em requisições ativas instantâneas. Provê balões textuais instanciados por coordenadas de mouse (Tooltip) sobre a camada Z-Index.

5. Infraestrutura e Roteiro Histórico de Sustentação

5.1. Log Forense de Mudanças de Paradigma (Changelog Resolutivo)

O ecossistema em produção hoje é reflexo direto de processos de estabilização contínua. As cinco fases críticas abaixo documentam anomalias nativas das infraestruturas de nuvem superadas no código em vigor:

  1. Fase 1: Eliminação de Vulnerabilidade Client-Side
    Anomalia Histórica: A arquitetura preliminar injetava a constante de configuração `firebaseConfig` diretamente na tag de inicialização, deixando o banco suscetível a scripts injetáveis pelo console do Chrome.
    Solução Permanente: Excisão sumária de todo o ecossistema Firestore do lado do navegador. Todo o fluxo de manipulação de dados foi transposto e centralizado nos Cloud Functions, convertendo as SPA's em clientes burros.
  2. Fase 2: Conflito de Variáveis no Identity Toolkit
    Anomalia Histórica: Durante a tentativa inicial de login via Fetch API na rota REST do Google, o servidor estourou recusa em virtude do uso de nome de variável proibido (`FIREBASE_API_KEY`) no ambiente de virtualização do Node.
    Solução Permanente: Renomeação e isolamento das chaves de autorização para propriedades anônimas independentes (IDENTITY_API_KEY) dentro do wrapper .env.
  3. Fase 3: Queda de CORS e Omissão de Banco de Dados
    Anomalia Histórica: A aplicação de JWT forçou o envio de Headers não-padrão no fluxo HTTP, causando erro de Preflight OPTIONS (401/403) nas portas do Google Cloud Run. Simultaneamente, painéis renderizavam vazios porque a requisição de busca orderBy('timestamp') omitia as centenas de reservas anteriores à migração.
    Solução Permanente: Atribuição de configuração allUsers nas invocações do Cloud Run aliada a hardcoding no Node retornando Status 204 No Content para tráfego em OPTIONS. Transferência da responsabilidade de cronologia do motor do Firebase para o V8-Engine da RAM do servidor.
  4. Fase 4: Dessincronização Visual Temporal (DST Bug)
    Anomalia Histórica: O lançamento de Feriados Integrais com demarcação `T00:00` sofria subtração automática pelo JavaScript do navegador baseado no fuso horário do aparelho, exibindo eventos de dia integral ocorrendo na noite do dia anterior.
    Solução Permanente: Refatoração completa dos iteradores que alimentam o grid dinâmico. O sistema passou a rejeitar heurísticas de fuso, analisando as Strings em formato primitivo de texto puro.

6. Protocolo de Deploy e Implantação

6.1. Otimização de Hospedagem (Firebase Hosting Multisite)

Tendo em vista a quantidade de portais desacoplados, a infraestrutura abandona o modelo de pasta única public/ e adota o roteamento avançado via arquivo firebase.json. Esta técnica mapeia "Targets" (alvos virtuais na nuvem) para subdiretórios individuais do seu repositório local, preservando as referências relativas do código sem a necessidade de reescrever links de imagens.

6.2. Diagrama de Árvore de Diretórios (Obrigatório)

📦 ROOT_PROJECT/
 ┣ 📂 app-acompanhamento/ (Index independente)
 ┣ 📂 app-calendario/ (Index independente)
 ┣ 📂 app-formulario/ (Index independente)
 ┣ 📂 app-manutencao/ (Index independente)
 ┣ 📂 app-painel/ (Index independente)
 ┣ 📂 functions/
 ┃ ┣ 📜 index.js
 ┃ ┣ 📜 package.json
 ┃ ┗ 📜 .env
 ┣ 📜 .firebaserc (Arquivo de mapeamento oculto)
 ┗ 📜 firebase.json (Arquivo de orquestração de deploy)

6.3. Operações via Linha de Comando (Firebase CLI)

Ao mudar de máquina de desenvolvimento, clone o repositório, abra o terminal e assegure que o vínculo dos targets na nuvem seja restabelecido apenas uma vez executando as rotinas de apply:

firebase target:apply hosting formulario-reservas NOME_ALVO_LÁ_NO_CONSOLE_1
firebase target:apply hosting painel NOME_ALVO_LÁ_NO_CONSOLE_2
firebase target:apply hosting calendario NOME_ALVO_LÁ_NO_CONSOLE_3
firebase target:apply hosting acompanhamento NOME_ALVO_LÁ_NO_CONSOLE_4

Para publicações contínuas e rotineiras de manutenção, utilize os empacotadores modulares abaixo visando implantações limpas e rápidas (na ordem de segundos):

// Envio global unificado (Frontend e Backend - Não Recomendado para rotina)
firebase deploy

// Atualização exclusiva de Regras de Negócio de Servidor
firebase deploy --only functions

// Atualização global de todas as frentes de Interface
firebase deploy --only hosting

// Atualização Modular (Específica - Altamente Recomendado)
firebase deploy --only hosting:painel
firebase deploy --only hosting:calendario
